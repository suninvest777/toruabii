# Google Search Console — настройка для 24toruabi.ee

## Почему в отчёте «страница с переадресацией»

Google **не индексирует** URL, которые только отдают **301/302** на другой адрес (например `http://` → `https://`). В индекс попадает **конечный HTTPS-URL**. В Search Console проверяйте именно **`https://24toruabi.ee/...`**, а не `http://`.

## Что сделать вручную в GSC

1. **Свойство:** используйте ресурс с префиксом **`https://24toruabi.ee`** (или доменное свойство с приоритетом HTTPS).

2. **Проверка URL:** вводите полный адрес с **`https://`** (например `https://24toruabi.ee/`).

3. **Файлы Sitemap:** раздел «Индексирование» → «Файлы Sitemap» → добавьте:
   ```text
   https://24toruabi.ee/sitemap.xml
   ```
   Статус «Успешно» может появиться через несколько дней.

4. **Связь sitemap с URL:** строка «Нет ссылающихся файлов Sitemap» для **http**-версии страницы — нормальна: в [robots.txt](../public/robots.txt) указан только HTTPS-sitemap.

## Технически на сайте

- В [robots.txt](../public/robots.txt): `Sitemap: https://24toruabi.ee/sitemap.xml`
- Редиректы в [src/middleware.ts](../src/middleware.ts): **HTTP→HTTPS** и **`www.24toruabi.ee` → `24toruabi.ee`** (один 301, если нужно и то и другое).

### DNS (Render)

Чтобы редирект `www` сработал, в панели DNS для домена должна быть запись **`www`** (CNAME на сервис Render или A на их IP), иначе запрос до приложения не дойдёт. Apex уже настроен для основного сайта.

### Sitemap (динамический)

Файл **`/sitemap.xml`** генерируется сервером из [`src/data/sitemap-entries.ts`](../src/data/sitemap-entries.ts) и [`src/pages/sitemap.xml.ts`](../src/pages/sitemap.xml.ts). При добавлении новых страниц обновляйте список в `sitemap-entries.ts`. Статический `public/sitemap.xml` не используется.

## «Обнаружена, не проиндексирована» (Discovered, not indexed)

В отчёте **Discovered – currently not indexed** / **«Обнаружена, не проиндексирована»** Google сообщает: URL **уже известен** (sitemap, внутренние ссылки), но страница **ещё не внесена в индекс** или индексация **отложена** (очередь обхода, оценка полезности).

Это **не то же самое**, что блокировка в `robots.txt` или `noindex`. Колонка «Последнее сканирование: отсутствует» часто значит, что обход по URL ещё не отражён в отчёте.

**Что делать:** запросить индексацию в GSC для важных URL, убедиться, что отправлен `sitemap.xml`, усилить внутренние ссылки на статьи блога, повторить проверку через несколько недель. Техническая разметка сайта (canonical, sitemap, отсутствие лишнего `noindex` на блоге) при этом может быть в порядке — отчёт отражает **решение поисковой системы**, а не обязательно ошибку в коде.

## «Вариант страницы с тегом canonical» и `?lang=ru`

В отчёте **«Страницы с альтернативным каноническим тегом»** / **«Вариант страницы с тегом canonical»** часто попадают URL вида `https://24toruabi.ee/faq?lang=ru`, `.../hinnakiri?lang=ru`, `.../blog/...?lang=ru`.

Это **не ошибка сайта**. В [`BaseLayout`](../src/layouts/BaseLayout.astro) канонический URL строится **без query-параметров**: для `/faq?lang=ru` в HTML указан `rel="canonical"` на `https://24toruabi.ee/faq`. Google **не индексирует** отдельно вариант с `?lang=ru`, чтобы не плодить дубли; в индексе остаётся одна страница на путь (каноническая без `?lang=ru`).

**Что делать в GSC:** ориентируйтесь на индексацию **канонических** URL без `?lang=ru`. Строки про `?lang=ru` в отчёте об исключениях можно считать ожидаемыми.

**Если понадобится отдельная индексация русской версии в Google** (отдельные сниппеты для RU), потребуется отдельная структура URL (например префикс `/ru/...`), согласованный hreflang и обновление sitemap — это **крупная доработка**, не входящая в текущую схему с переключателем `?lang=ru`.
